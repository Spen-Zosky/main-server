## 🎯 **ENHANCED CHAIN OF THOUGHT (CoT) DECISION PROCESS**

Tutti i membri del team devono adottare una strutturata Chain of Thought (CoT) per garantire rigore, consistenza e responsabilità in ogni decisione o azione del progetto.

### **📊 8-STEP DECISION MAKING PROCESS**

| Step | Domanda Guida | Obiettivo | Esempi Agent-Specifici |
|------|---------------|-----------|------------------------|
| **1. Context** | In che contesto operativo sto intervenendo? | Inquadrare livello, ambiente, vincoli | **Solution Architect**: Design di sistema per scaling produzione<br>**Fullstack Developer**: Integrazione feature frontend + backend<br>**Data Architect**: Ottimizzazione database in ambiente produzione |
| **2. Objective** | Qual è lo scopo preciso della mia azione? | Allineare intervento a obiettivo concreto | **Solution Architect**: Design architettura scalabile per crescita 10x<br>**Backend Developer**: Refactoring endpoint API per migliore manutenibilità<br>**Frontend Developer**: Creazione libreria componenti riutilizzabili |
| **3. Relevance** | Perché è importante farlo ora? È prioritario? | Valutare urgenza, impatto, coerenza roadmap | **Solution Architect**: Decisione architetturale blocca progresso team<br>**Fullstack Developer**: Vulnerabilità sicurezza richiede fix immediato<br>**Data Architect**: Performance database impatta esperienza utente |
| **4. Impacts** | Chi o cosa sarà influenzato dalla mia decisione? | Considerare effetti su codice, dati, utenti | **Solution Architect**: Tutti i team di sviluppo, scalabilità sistema<br>**Backend Developer**: Consumer API, integrità dati, affidabilità sistema<br>**Frontend Developer**: Interfaccia utente, accessibilità, compatibilità mobile |
| **5. Alternatives** | Ci sono opzioni diverse? Quali sono i trade-off? | Valutare soluzioni, costi/benefici | **Solution Architect**: Microservizi vs monolito - complessità vs performance<br>**Backend Developer**: REST vs GraphQL - semplicità vs flessibilità query<br>**Frontend Developer**: React vs Vue - ecosistema vs curva apprendimento |
| **6. Evidence** | Ho elementi oggettivi per decidere? Ho verificato? | Basare azione su dati, log, test | **Solution Architect**: Benchmark performance, risultati load testing<br>**Backend Developer**: Tempi risposta API, log errori, performance database<br>**Frontend Developer**: Analytics utente, audit accessibilità, testing cross-browser |
| **7. Communication** | Devo informare o coordinare con altri? | Assicurare trasparenza e collaborazione | **Solution Architect**: Notificare tech lead, aggiornare architecture board<br>**Backend Developer**: Notificare consumer API, coordinare con team frontend<br>**Frontend Developer**: Aggiornare team design system, informare QA tester |
| **8. Documentation** | Devo registrare l'azione? Dove e come? | Assicurare auditabilità e condivisione knowledge | **Solution Architect**: Aggiornare architecture decision records (ADR)<br>**Backend Developer**: Aggiornare documentazione API, creare changelog<br>**Frontend Developer**: Aggiornare documentazione componenti, creare style guide |

## 🔍 **WEB VALIDATION PROTOCOL**

### **🚨 DIRETTIVA CRITICA - VALIDAZIONE WEB**

**SEMPRE utilizzare validazione browser-based invece di semplici curl check:**

```bash
# ❌ MAI usare solo questo:
curl http://localhost:5173

# ✅ SEMPRE usare questo per verifica web:
npm run validate:service http://localhost:5173

# ✅ O per check multipli:
npm run validate:oci localhost

# ✅ O per solo browser se necessario:
npm run validate:browser http://localhost:5173
```

**RAZIONALE**: Sistema headless può avere:
- Errori JavaScript
- Problemi caricamento asset
- Issues CORS
- Problemi routing frontend
- Issues di performance

### **🛠️ VALIDATION COMMANDS**

```bash
# Full service validation (browser + curl)
npm run check <url>

# Check all services on host
npm run validate:oci <host>

# Browser-only validation
npm run validate:browser <url>

# Quick curl check (solo API endpoints)
npm run validate:curl <url>
```
